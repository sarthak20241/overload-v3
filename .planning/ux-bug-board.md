7-lens audit of the workout experience. 22 verified bugs + 32 UX improvements.

## Verified bugs
### [HIGH/M] Editing a unilateral set only changes the left side; right side silently kept stale
- lens: set-types-rpe-unilateral | area: Edit past workout (app/workout/edit/[id].tsx)
- detail: The editor renders only `s.weight`/`s.reps` (the left side) in the set-row inputs (edit/[id].tsx:624-644) while `reps_right`/`weight_kg_right`/`rpe_right` round-trip untouched (saved as-is at :383-385). A set logged as '40/35 x 8/7' displays as just '40' and '8'. If the user edits that weight to 42, the set is saved as left=42, right=35 (unchanged) — an inconsistent per-side set the user never saw and cannot inspect or correct. There is also no badge, so they have no idea the set is unilateral. Trigger: log a unilateral set with different per-side loads, then open the workout in edit mode and change the weight.
- impact: Silent data corruption on a past set, with the second side invisible. The user believes they fixed a set's weight but only half of it changed, and they can't see or repair the other half.
- fix: In app/workout/edit/[id].tsx the editor renders only the left side (s.weight/s.reps at lines 624-644) of unilateral sets while reps_right/rpe_right/weight_kg_right round-trip untouched (saved as-is at 382-385). Editing the weight/reps of a unilateral set silently diverges the two sides (e.g. 40/35 → edit 40 to 42 → saves 42/35), and there is no unilateral badge so the user has no idea the value is one of two. setVolumeKg (lib/sets.ts:25) consumes weight_kg_right*reps_right, so this corrupts volume/PRs, and history.tsx:68 surfaces the divergence the user never created.

Preferred fix (M): for sets where s.is_unilateral, render an L/R pair of inputs for each weight/reps axis (bind right inputs to weight_kg_right/reps_right, parse them in handleSave instead of copying through), plus a small unilateral badge on the row so the user understands the set has two sides. Reuse the existing per-side display logic from history.tsx (uniWeightDiffers / the 40kg×8 / 35kg×7 formatter) for consistency.

Minimum fix (S): show a unilateral badge on the row and, when a unilateral set's weight or reps is edited, keep the two sides in sync (write the new left value into weight_kg_right/reps_right too) rather than letting them silently diverge — this at least prevents creating a per-side split the user never intended, at the cost of flattening any genuine per-side difference (acceptable until the full L/R UI ships). Do NOT leave the current behavior of editing left while keeping a stale right.

### [HIGH/M] "Tap to split" on a 3+ giant set ejects a member instead of splitting the group
- lens: supersets-experience | area: Routine editor — superset link/unlink (app/(app)/routines.tsx, toggleSupersetLink L576-591 + normalizeGroups L135-153)
- detail: In the editor, a superset of 3+ members renders a "Superset — tap to split" divider above every member. The user reads each divider as a cut point: tapping it should split into two contiguous groups. But toggleSupersetLink only ever nulls the single row i (next[i].supersetGroup = null); normalizeGroups then re-coalesces the rest. For a group [A,B,C,D] (all group 1): tapping the divider between A and B (i=1) yields [null,null,1,1] — A AND B both become solo and only C,D stay grouped (B was silently ejected). Tapping between B and C (i=2) yields [1,1,null,null] — C is ejected, only A,B survive. Only the LAST divider (i=3, ABC|D) behaves as a clean split. Verified by simulating normalizeGroups+toggle for 3- and 4-member groups. For a 2-member group it happens to look correct (ejecting one leaves the other solo). So any attempt to split a giant set in the middle quietly destroys part of the superset with no warning, and the user has to rebuild it link by link.
- impact: A user who builds a 3+ exercise giant set and later wants to break it into two supersets (or peel off the first movement) will instead lose a different member than the one they tapped, with no feedback. The structure they carefully built silently degrades, and the only recovery is to re-link from scratch.
- fix: Make the divider a true cut, not a single-row null. In toggleSupersetLink (app/(app)/routines.tsx L576-591), when the row is linkedAbove (the split case), walk from i to the end of the current contiguous group and assign that tail a fresh temporary id, leaving rows [firstMember..i-1] as the original group; then run normalizeGroups so [A,B,C,D] split at i=2 becomes A,B | C,D and split at i=1 becomes A | B,C,D. Concretely: let end=i; while(end<next.length && next[end].supersetGroup===next[i].supersetGroup) end++; then for(k=i;k<end;k++) next[k].supersetGroup = 9000+i (a temp id distinct from the original), before returning normalizeGroups(next). The unlink-the-last-pair case still collapses to solo correctly because a 2-run that loses one member normalizes to solo. Add a unit test that splits at every divider of a 4-member group and asserts two contiguous groups result (no ejected members).
